A Hybrid NLP–LLM Pipeline for End-to-End Requirements Engineering

Abstract—Requirements defects, as a root cause of software project failure, have received a remarkable amount of attention relative to how little the situation has actually improved. Industry surveys going back twenty-plus years consistently attribute somewhere around 40% of project failures to poorly written or entirely missing requirements specifications [1], and yet the process by which stakeholder interviews get translated into unambiguous, verifiable documents has not changed in any fundamental sense for most organizations. This paper describes AutoReq, a pipeline we developed to try to automate this translation for Turkish-language input—we chose Turkish specifically because most existing tools simply do not support it, which is a gap that is rarely acknowledged. The system works in three stages: a Stanza-based preprocessing module handles sentence segmentation, actor and business object extraction, and FR/NFR labeling; three LLM modules (ConflictDetector, GapAnalyzer, RequirementImprover) run in parallel over Gemini 2.5 Flash via OpenRouter and return structured JSON; a generation layer at the end assembles an ISO/IEC/IEEE 29148-compliant SRS, user stories, Gherkin BDD scenarios, and a product backlog. Results on a 63-sentence development corpus covering five domains showed 56.1% precision and 82.2% recall for actor extraction, 90.5% FR/NFR classification accuracy, and 100.0% contradiction detection with 0% false positives. On a separately held-out 30-sentence healthcare corpus, FR/NFR accuracy reached 96.7% and actor recall 90.5%. Expert raters scored the generated SRS documents at an average of 4.34 out of 5. The numbers are reasonable for a first system of this kind, though we would not want to overstate them.

Keywords—Software Requirements Engineering, Natural Language Processing, Large Language Model, Automated Document Generation, Software Requirements Specification, Contradiction Detection, Behavior-Driven Development, Requirements Classification

I. INTRODUCTION

Software projects fail in many ways, and we are not going to pretend that requirements are the only problem, but they do appear in post-mortems with a consistency that is somewhat difficult to ignore. The Standish Group's CHAOS Report puts requirements-related failures at around 40% of all project failures [1]—a figure that has been cited so many times it has almost lost its impact, but the underlying observation is still valid. Boehm and Basili [2] showed, back in 2001, that catching a defect during the requirements phase costs roughly one tenth of what the same defect costs if it reaches implementation, and substantially less than what it costs if it reaches production. These numbers have been known for a long time. The practice has not changed much.

Part of why this is so persistent, we think, is structural. Contradictions in a requirements document rarely appear as obvious statements of the form "X must happen and X must not happen" in adjacent sentences. They tend to emerge from interactions between requirements written at different times, often by different stakeholders, responding to different pressures. A requirement saying "the system should be fast" can pass review without anyone flagging it, and then cause real problems when developers sit down to implement it and realize no one specified what fast means in this context [3]. Wagner et al. [3] looked at this pattern across multiple countries and multiple organization types and found the same thing everywhere—it is not a sign of immature teams, it is a structural property of the elicitation process itself, which is actually a somewhat uncomfortable conclusion for the field.

Various tools have tried to address portions of this. Rule-based NLP handles entity extraction reasonably well under constrained conditions. Classification models have been applied to FR/NFR separation with mixed results. LLM-based systems can rewrite ambiguous requirements, sometimes convincingly, though they are not generally designed to detect logical contradictions across a full requirement set or enforce any specific document structure. To our knowledge—and we acknowledge we may have missed something—no currently published tool takes unstructured stakeholder text as input and produces ISO/IEC/IEEE 29148-compliant output in a single automated pass, without requiring manual intervention somewhere in the middle [4]. AutoReq was built to address that.

The system passes stakeholder text through three layers: morphological analysis and dependency parsing via Stanza, concurrent LLM analysis using Gemini 2.5 Flash over OpenRouter with JSON-constrained outputs, and a generation layer that produces four artifact types from a single analysis pass. We describe the architecture, the evaluation methodology, and the results in the sections that follow.

What makes AutoReq different from prior work, at least in our view, comes down to a few things. The NLP–LLM integration handles the entire pipeline without any manual handoffs between stages, which sounds straightforward but turns out to require more engineering effort than one might expect. The ConflictDetector does not just flag which requirements conflict—it records what kind of conflict is present and why, in natural language, which is the information an analyst actually needs to do something with the finding. And all four output artifact types are produced from a single analysis pass, rather than requiring separate processing runs for each format.

The remainder of the paper is organized as follows. Section II surveys related work. Section III describes the architecture. Section IV reports evaluation results. Section V draws conclusions.

II. LITERATURE REVIEW

Automated extraction of requirements from natural language text is not a new problem. It has been studied within NLP for quite some time, with varying degrees of success and, one has to say, varying degrees of practical applicability.

Dalpiaz et al. [6] combined interpretable machine learning with dependency parse features and reported FR/NFR classification accuracy above 85%, which is good. The caveat is that their approach was designed for English, and it does not transfer well to Turkish. Turkish is agglutinative—a single root can appear in dozens of inflected surface forms depending on morphological context, which breaks the feature assumptions underlying most English-trained classifiers in ways that are not easy to patch. Ferrari and Esuli [7] tackled ambiguity at the lexical level across domains, which is a useful contribution, but their method does not deal with contradictions that only emerge when two non-adjacent sentences are read against each other. That particular problem has received less attention than it probably deserves.

LLM-based approaches to software engineering tasks have proliferated, though not evenly. Fan et al. [8] note in their survey that code generation and bug localization dominate the literature, while requirements engineering specifically remains underexplored. Ronanki et al. [9] examined LLM-assisted elicitation and found a pattern that, once you see it, you see it everywhere: the models generate plausible-sounding clarifying questions but also introduce hallucinated requirements and structurally inconsistent outputs with some regularity. The implication is straightforward—an LLM used without structured preprocessing and output validation is not a reliable component for requirements work. It can be useful inside a more constrained pipeline, but it should not be the pipeline.

Automated SRS generation, as a standalone research problem, has received surprisingly sparse attention, given how much effort the standards process has put into specifying what these documents should contain. ISO/IEC/IEEE 29148 [4] lists the required sections—scope statements, stakeholder definitions, use cases, formal requirement tables—but most tools that claim to produce SRS output either fill those sections from rigid templates or depend on analysts entering content manually. We could not find a published system that derives section content automatically from an intermediate analytical representation of the requirements. There may be commercial tools that do something like this, but if so, they are not described in any accessible publication.

Contradiction detection has a longer history, grounded originally in formal logic and ontology-based reasoning. Guo et al. [10] report 83.9% accuracy on pairwise conflict detection using semantic analysis. Their system identifies conflicting pairs, which is useful, but it assigns the same generic conflict label regardless of whether the problem is a direct logical contradiction, a scope overlap, or an implied missing precondition. For someone trying to actually fix the conflict, those distinctions matter considerably—the corrective actions are quite different.

BDD scenario generation from requirements is a more recent area. Ronanki et al. [11] evaluated ChatGPT specifically on user story quality and found substantial variation in output consistency; the generated stories needed expert review before anyone would trust them in a real workflow. This is, again, consistent with the elicitation findings—output quality downstream is always bounded by how well the upstream extraction step worked, and this seems to be a constraint that is underappreciated in a lot of the recent LLM-in-SE literature.

The overall picture, then, is a field where individual pipeline components have been studied, sometimes quite thoroughly, but where no integrated system connecting extraction, classification, contradiction detection, and scenario generation has been published. AutoReq is an attempt to build that integration, with the added constraint of supporting Turkish, which is somewhat non-trivial given the language's morphological complexity. III. SYSTEM ARCHITECTURE AND METHODOLOGY

AutoReq is organized into three sequential layers. Raw stakeholder text enters a preprocessing layer, the output is passed to an LLM analysis layer, and the results are then handed to a generation layer that assembles the final documents. One of the early design decisions that saved us a lot of debugging effort was enforcing that all data exchange between layers happen exclusively through two typed classes defined in core/models.py: ParsedDocument for preprocessing output, and AnalysisReport for LLM analysis output. A field cannot appear as input to a downstream layer unless an upstream layer has explicitly written it to the appropriate class. The whole system is implemented in Python; users interact through a Streamlit application that calls a central process_text() function.

fig 1

Preprocessing. The preprocessing layer runs the Turkish Stanza model [5], applying tokenization, multi-word tokenization (MWT), POS tagging, lemmatization, and NER to each input text. Memory consumption is approximately 150 MB; @st.cache_resource prevents the model from being reloaded on repeated calls. Sentence boundary detection uses Stanza's built-in segmenter, and each sentence is assigned a unique identifier in the REQ_001 format.

Actor extraction works via lookup against a predefined lemma table. If a sentence's lemma sequence contains any entry from the set {kullanıcı, sistem, yönetici, admin, müşteri}, the corresponding actor is recorded; business objects use an analogous table. Working at the lemma level is, frankly, not optional for Turkish—surface-level matching would produce counts that are meaninglessly inflated given the morphological variation. When Stanza fails to initialize, which happened more often than we would have liked during development, a substring-matching fallback handles extraction instead. After initial testing revealed coverage gaps, we extended the lemma table to include roles common in healthcare, education, logistics, and legal domains, and added a secondary extraction pass using Stanza dependency trees, treating nsubj-tagged nouns as additional actor candidates while filtering out system-level tokens like 'uygulama' and 'platform' when a human role was already present in the sentence.

FR/NFR classification uses keyword matching: a requirement is labeled NON_FUNCTIONAL if its text contains a term from a predefined indicator list (performance, security, latency, SSL, and related terms); everything else receives the FUNCTIONAL label. This worked adequately for most of the development data. It did not work well for sentences involving kullanıcı and şifre—both terms ended up in the non-functional keyword list despite appearing constantly in functional requirements, and the false positive NFR labels this generated account for a disproportionate share of the classification errors we report in Section IV. Replacing the keyword matcher with a properly trained classifier is the obvious fix; we discuss this in the limitations.

LLM Analysis. The ParsedDocument is forwarded concurrently to three LLM modules running inside a ThreadPoolExecutor: ConflictDetector, GapAnalyzer, and RequirementImprover. Each module serializes the requirement list in the format [REQ_NNN] (TYPE) TEXT, wraps it in a two-part prompt consisting of a system message defining the module's role and a user message specifying the requirement list and expected JSON schema, and submits it to the Gemini 2.5 Flash API. The API occasionally returns markdown-formatted text rather than bare JSON—an annoyance we did not anticipate initially—so all three modules pass their responses through an extract_json_object() utility that strips formatting before parsing.

fig 2

ConflictDetector classifies conflicts into three types: direct logical opposition (two requirements that are mutually exclusive—one requiring password reset and another prohibiting password changes, for instance), scope overlap (the same operation specified with incompatible parameters in two different requirements), and missing critical processes (functional flows referenced but never fully defined, such as an authentication requirement with no corresponding account-recovery path). GapAnalyzer checks for flows implied by the requirement set but not actually stated in it—a login flow with no error-handling requirement is a typical example. RequirementImprover rewrites underspecified statements to include measurable criteria; "the system should respond quickly" becomes "the system shall respond within 1 second of form submission." All three modules attach a severity rating of low, medium, or high to their findings, and these ratings propagate into artifact prioritization downstream.

Output Generation. The three LLM modules write their results to an AnalysisReport, which the generation layer then reads to produce four output types.

The primary output is an SRS structured according to ISO/IEC/IEEE 29148 [4], with four main sections—Introduction, Overall Description, Specific Requirements, and Verification—populated from AnalysisReport fields. PDF output uses fpdf2, which handles Turkish characters correctly without additional configuration, which was our main selection criterion; DOCX output uses python-docx. Each requirement in the Specific Requirements section gets its own table row with identifier, classification label, actors, business objects, and the improved text from RequirementImprover.

User stories are generated for functional requirements using the "As a [actor], I want to [action], so that [benefit]" template, with the actor field from NER output, the action from dependency parse results, and the benefit clause from LLM improvement records. Priority matches the severity rating of any linked analysis finding.

fig 3

Gherkin BDD scenarios are derived from the user stories, with each story expanding into a Given/When/Then block. Preconditions, inputs, and acceptance criteria come from the improvement records produced earlier, so the measurable criteria introduced during LLM analysis carry through directly into test scenarios. The output .feature files are formatted for direct import into standard test automation frameworks. The fourth output is a product backlog in XLSX format, with each item inheriting its priority from the severity rating of the corresponding user story, preserving traceability from the original requirement through to the sprint artifact.

IV. RESULTS AND EVALUATION

fig 4

Evaluation used two corpora. The development corpus comprised 63 sentences across five domains—e-commerce, online banking, appointment scheduling, corporate HR, and mobile application. A separate 30-sentence healthcare corpus was held out from all development and optimization work and used only for generalization testing. Both corpora were adapted from real-world stakeholder materials. Annotation was carried out by the primary author and cross-validated using Gemini and Claude as independent labelers, with disagreements resolved by author judgment—an imperfect approach but a reasonable one given the absence of established Turkish requirements benchmarks. All experiments used the same hardware and Gemini 2.5 Flash via OpenRouter.

Development corpus. Actor extraction yielded 56.1% precision and 82.2% recall (F1=66.7%). The precision gap is driven by false positives: the system tags incidental mentions of actors (e.g. "hasta" in "hasta verilerinin yedeklenmesi") as extracted actors, inflating the count. Recall improved substantially after extending the lemma table and adding dependency-parse-based extraction. FR/NFR classification reached 90.5%, up from the earlier 69.8%, reflecting the three-layer hybrid classifier (FR verb detection → NFR keyword+regex → LLM fallback) that replaced the original keyword-only approach.

ConflictDetector identified 100.0% of ground-truth conflict pairs (6/6) with a 0.0% false positive rate—a substantial improvement over the earlier 85.7% detection at 11.3% false positives. The improvement is attributable to confidence-based post-processing (threshold=0.6) and pairwise deduplication filtering that were added after the initial evaluation. Expert ratings of the generated SRS documents averaged 4.34 out of 5 for completeness and structural readability, across three independent software engineering researchers.

Healthcare generalization. On the held-out healthcare corpus, FR/NFR accuracy came in at 96.7%, actor precision at 51.4%, and actor recall at 90.5% (F1=65.5%). The precision drop relative to the development corpus is expected: domain-specific terms like "hasta" appear both as actors and as modifiers in non-actor contexts (e.g. "hasta verilerinin yedeklenmesi"), generating false positives that the lemma-based system does not filter. The high recall, however, confirms that the extended lemma table and dependency-parse extraction generalize well to unseen healthcare vocabulary. FR/NFR classification improved markedly over the development corpus, suggesting that the three-layer classifier handles domain-specific NFR patterns (KVKK compliance, SLA targets, medical data encryption) robustly.

Ablation. Disabling all three LLM modules drops conflict detection to 0%, which confirms the obvious point that the NLP layer alone cannot perform logical analysis across a requirement set. The hybrid system adds roughly 11.8 seconds per sentence on average (NLP-only: 0.84s, hybrid: 12.64s), with a standard deviation of 3.03s that reflects API latency rather than any internal variance. The higher per-sentence hybrid latency compared to earlier measurements reflects the addition of pairwise conflict analysis and parallel gap detection per sentence. For single-document use, this is not a meaningful obstacle. Batch processing is another matter and would benefit from response caching or asynchronous calls; the parallel thread structure already in place helps somewhat.

A few limitations bear mentioning. The 63-sentence development corpus is small, and the reported figures should not be read as definitive performance estimates—they are indicative. Larger evaluations using something like the PROMISE dataset or a relevant OpenScience subset would give a more reliable picture. The system supports only Turkish; extending to other languages would require Stanza language models and rebuilt lexical resources for each. The FR/NFR keyword classifier has a known failure mode that a trained classifier would address; this is next on the list.

V. CONCLUSION

AutoReq is, to our knowledge, the first end-to-end pipeline that takes Turkish stakeholder text as input and produces a complete set of requirements engineering artifacts—SRS, user stories, Gherkin scenarios, product backlog—without manual intervention between stages. The hybrid architecture connects Stanza-based morphological preprocessing to three concurrent LLM analysis modules, with a generation layer that handles all four output types in a single pass. On the development corpus we obtained 100.0% contradiction detection with 0% false positives, 82.2% actor recall, and 90.5% FR/NFR classification. On the held-out healthcare corpus, FR/NFR accuracy reached 96.7% and actor recall 90.5%, which suggests both the NLP and LLM layers generalize well beyond the development domains.

The system makes a few contributions worth summarizing. The most important is probably the conflict detection component itself—not because conflict detection is new, but because recording conflict type, requirement identifiers, and a natural-language rationale for each detected pair actually gives analysts something useful to work with, rather than a list of flagged pairs to investigate from scratch. The unified generation layer is also meaningful from a practical standpoint; producing four artifact types from a single analysis pass removes a class of integration headaches that tends to accumulate in multi-tool pipelines.

There are real limitations. The language-specific NLP configuration does not port easily to other languages. The external API calls introduce latency that will become a problem at batch scale, though the parallel thread structure mitigates this somewhat. The evaluation corpus is small enough that the numbers carry meaningful uncertainty—we have tried to be clear about this throughout.

Future plans include training a domain-specific Turkish NER model to replace the lemma-based approach, extending support to English and German, integrating with Jira and Azure DevOps, and running usability studies with actual requirements analysts to see how the system affects their workflows in practice. That last item is probably the most important one, and the one that tends to get postponed the longest.
