A Hybrid NLP–LLM Pipeline for End-to-End Requirements Engineering

Abstract—Requirements defects, as a root cause of software project failure, have received a remarkable amount of attention relative to how little the situation has actually improved. Industry surveys going back twenty-plus years consistently attribute somewhere around 40% of project failures to poorly written or entirely missing requirements specifications, and yet the process by which stakeholder interviews get translated into unambiguous, verifiable documents has not changed in any fundamental sense for most organizations. This paper describes AutoReq, a pipeline we developed to try to automate this translation for Turkish-language input—we chose Turkish specifically because most existing tools simply do not support it, which is a gap that is rarely acknowledged. The system works in three stages. A Stanza-based preprocessing module handles sentence segmentation, actor and business object extraction, and FR/NFR labeling. Two analysis modules, ConflictDetector and GapAnalyzer, run in parallel against Gemini 2.5 Flash through OpenRouter and return structured JSON, and a third module, RequirementImprover, kicks in later alongside the artifact generators in a second parallel stage. A generation layer at the end assembles an ISO/IEC/IEEE 29148-compliant SRS, user stories, Gherkin BDD scenarios, and a product backlog. Results on a 63-sentence development corpus covering five domains showed 56.1% precision and 82.2% recall for actor extraction, 93.7% FR/NFR classification accuracy, and 100.0% contradiction detection with 0% false positives. On a separately held-out 30-sentence healthcare corpus, FR/NFR accuracy reached 96.7% and actor recall 90.5%. The numbers are reasonable for a first system of this kind, though we would not want to overstate them.

Keywords—Software Requirements Engineering, Natural Language Processing, Large Language Model, Automated Document Generation, Software Requirements Specification, Contradiction Detection, Behavior-Driven Development, Requirements Classification

I. INTRODUCTION

Software projects fail in many ways, and we are not going to pretend that requirements are the only problem, but they do appear in post-mortems with a consistency that is somewhat difficult to ignore. The Standish Group's CHAOS Report puts requirements-related failures at around 40% of all project failures [1]—a figure that has been cited so many times it has almost lost its impact, but the underlying observation is still valid. Boehm and Basili [2] showed, back in 2001, that catching a defect during the requirements phase costs roughly one tenth of what the same defect costs if it reaches implementation, and substantially less than what it costs if it reaches production. These numbers have been known for a long time. The practice has not changed much.

Part of why this remains so persistent is structural. Contradictions in a requirements document rarely look like "X must happen and X must not happen" sitting next to each other. They emerge from interactions between requirements written at different times, often by different people responding to different pressures. A requirement saying "the system should be fast" passes review without anyone flagging it—and then causes real problems when developers sit down to implement it and realize nobody ever said what fast actually means in this context [3]. Wagner et al. [3] looked at this pattern across multiple countries and organization types and found it everywhere. It is not a sign of immature teams. It is a structural property of the elicitation process itself—which is, honestly, not a comfortable thing for the field to sit with.

Various tools have tried to address portions of this. Rule-based NLP handles entity extraction reasonably well under constrained conditions. Classification models have been applied to FR/NFR separation with mixed results. LLM-based systems can rewrite ambiguous requirements, sometimes convincingly, though they are not generally designed to detect logical contradictions across a full requirement set or enforce any specific document structure. To our knowledge—and we acknowledge we may have missed something—no currently published tool takes unstructured stakeholder text as input and produces ISO/IEC/IEEE 29148-compliant output in a single automated pass, without requiring manual intervention somewhere in the middle [4]. AutoReq was built to address that.

The system passes stakeholder text through three layers: morphological analysis and dependency parsing via Stanza, concurrent LLM analysis using Gemini 2.5 Flash over OpenRouter with JSON-constrained outputs, and a generation layer that produces four artifact types from a single analysis pass. We describe the architecture, the evaluation methodology, and the results in the sections that follow.

What makes AutoReq different from prior work, at least in our view, comes down to a few things. The NLP–LLM integration handles the entire pipeline without any manual handoffs between stages, which sounds straightforward but turns out to require more engineering effort than one might expect. The ConflictDetector does not just flag which requirements conflict—it records what kind of conflict is present and why, in natural language, which is the information an analyst actually needs to do something with the finding. All four output artifact types come out of a single analysis pass—no separate processing runs for each format.

The rest of the paper covers related work in Section II, system architecture in Section III, evaluation results in Section IV, and conclusions in Section V.

II. LITERATURE REVIEW

Automated extraction of requirements from natural language text is not a new problem. It has been studied within NLP for quite some time, with varying degrees of success and, one has to say, varying degrees of practical applicability.

Dalpiaz et al. [6] combined interpretable machine learning with dependency parse features and reported FR/NFR classification accuracy above 85%, which is good. The caveat is that their approach was designed for English, and it does not transfer well to Turkish. Turkish is agglutinative—a single root can appear in dozens of inflected surface forms depending on morphological context, which breaks the feature assumptions underlying most English-trained classifiers in ways that are not easy to patch. Ferrari and Esuli [7] tackled ambiguity at the lexical level across domains, which is a useful contribution, but their method does not deal with contradictions that only emerge when two non-adjacent sentences are read against each other. That particular problem has received less attention than it probably deserves.

LLM-based approaches to software engineering have become widespread, but that momentum hasn't reached every subfield equally. Fan et al. [8] point out that while code generation and error localization dominate the literature, requirements engineering remains largely underexplored.

Ronanki et al. [9] looked at LLM-assisted information retrieval and found a recurring problem: models generate questions that sound plausible, but they also produce fabricated requirements and structurally inconsistent outputs. The pattern is consistent enough to draw a clear conclusion. An LLM without structured preprocessing and output validation is not a reliable component for requirements engineering. It can play a role in a more constrained workflow, but it shouldn't become the workflow itself.

Automated SRS generation has received surprisingly little attention as a standalone research problem, especially given how much effort has gone into defining what these documents should contain. ISO/IEC/IEEE 29148 [4] specifies the required sections: scope statements, stakeholder definitions, use cases, and formal requirement tables. But most tools that claim to produce SRS output either populate those sections from rigid templates or require analysts to enter content manually.

We found no published system that derives section content automatically from an intermediate analytical representation of the requirements. There may be commercial tools that do this. If so, they are not described in any source we could access.

Contradiction detection has roots in formal logic and ontology-based reasoning, so there's more history here than in most other areas. Guo et al. [10] report 83.9% accuracy on pairwise conflict detection using semantic analysis. Identifying conflicting pairs is genuinely useful. The limitation is the labeling: every conflict gets the same generic tag, whether the underlying issue is a direct logical contradiction, a scope overlap, or a missing precondition. Those are not the same problem. The corrective actions for each are completely different, and a single label doesn't help anyone figure out which one they're dealing with.

Ronanki et al. [11] tested ChatGPT on user story quality. Output varied enough that expert review was mandatory before anything reached a real workflow — not optional, mandatory. That tracks with what the elicitation work shows. Downstream quality doesn't exist independently of upstream extraction. Mess up the first step and everything after it carries that forward. A lot of recent LLM-in-SE work seems to miss this, treating pipeline stages like they're isolated from each other.

Across the literature, individual components get studied. Sometimes carefully. What hasn't been published is a system that connects extraction, classification, contradiction detection, and scenario generation into one pipeline. AutoReq is that attempt. Turkish support adds another layer — the morphology is complex enough that NLP tooling built around English tends to produce unreliable results, sometimes quietly, which is its own problem.

III. SYSTEM ARCHITECTURE AND METHODOLOGY

AutoReq has three layers. Stakeholder text enters at the top, passes through preprocessing, moves to LLM analysis, and the output from that reaches the generation layer, which produces the final documents.

Early on we decided that all data moving between layers had to go through exactly two typed classes in core/models.py. ParsedDocument carries preprocessing output. AnalysisReport carries LLM analysis output. Nothing reaches a downstream layer unless something upstream explicitly wrote it there. That constraint caught a surprising number of integration bugs early — probably more than any other single decision we made.

The system runs in Python. The Streamlit front-end calls a central process_text() function and that's the only entry point into the pipeline.

Fig. 1. Three-layer pipeline architecture of AutoReq.

The preprocessing layer runs the Turkish Stanza model [5] across each input text — tokenization, MWT, POS tagging, lemmatization, dependency parsing, NER. Memory footprint lands at around 300 MB, so @st.cache_resource keeps the model loaded between calls instead of reloading it every time. Sentence boundaries come from Stanza's built-in segmenter. Each sentence gets a unique identifier in REQ_001 format.

Actor extraction works off a predefined lemma table. The initial seed was generic, kullanıcı, sistem, yönetici, admin, müşteri, and if a sentence's lemma sequence hits any entry in the table, that actor gets logged. Business objects work the same way, just against a separate table.

Working at the lemma level isn't optional for Turkish. Surface-level matching produces inflated counts that don't mean anything, given how much morphological variation the language has.

Stanza failing to initialize was a bigger problem during development than we expected. A substring-matching fallback handles extraction when that happens.

Early testing exposed coverage gaps, so we grew the lemma table to more than sixty entries, picking up roles common in healthcare, education, logistics, and legal work (hasta, doktor, hemşire, öğrenci, avukat, pilot, kurye, koordinatör, and so on), plus eighteen multi-word actor patterns. There's also a secondary extraction pass using Stanza dependency trees — nsubj-tagged nouns come in as additional actor candidates, though system-level tokens like 'uygulama' and 'platform' get filtered out when a human role is already present in the sentence.

Classification runs through three layers. First, a functional-verb detector looks for action-oriented predicates. An ambiguous-keyword set sits next to it (güvenlik, yedek, erişilebilir, yük, hızlı, performans), and on purpose those words alone are not enough to kick a sentence out of Layer 1. Next is an NFR keyword-and-regex matcher: performance, security, latency, SSL, and related terms, plus seven numerical patterns for things like percentage thresholds, millisecond bounds, and concurrency limits. A substring-disambiguation regex keeps güvenli and güvenlik from matching in incidental contexts. Anything the first two layers don't resolve goes to Gemini for the final call.

The keyword-only baseline had a consistent failure mode. Terms like kullanıcı and şifre kept triggering NON_FUNCTIONAL labels despite appearing constantly in functional contexts. That alone dragged accuracy down to 69.8%. The hybrid system brings it to 93.7%.

The ParsedDocument goes first to two LLM modules running side by side in a ThreadPoolExecutor with max_workers=2: ConflictDetector and GapAnalyzer. RequirementImprover comes later, in a second ThreadPoolExecutor with max_workers=6, where it shares the pool with the SRS, user story, Gherkin, and backlog generators. Each LLM module serializes the requirement list in [REQ_NNN] (TYPE) TEXT format, wraps it in a two-part prompt — a system message defining the module's role and a user message specifying the requirement list and expected JSON schema — and sends it to the Gemini 2.5 Flash API.

The API occasionally returns markdown-formatted text instead of bare JSON. We didn't anticipate that, and it took more time to handle than it should have. All three modules now pass responses through an extract_json_object() utility that strips formatting before parsing.

Fig. 2. Example input–output of the LLM analysis layer: conflict detection and optimization results.

ConflictDetector sorts each detected pair into one of six buckets: logic, business_rule, performance, security, usability, or other. Logic is straight contradiction, one requirement asking for a password reset while another forbids password changes. business_rule covers two requirements that hit the same workflow with incompatible parameters. performance, security, and usability narrow conflicts down to those specific non-functional areas. Think contradictory latency targets, authentication policies that don't line up, or accessibility constraints that fight each other. If a pair doesn't fit any of those five, it lands in other.

GapAnalyzer looks for flows the requirement set implies but never states. A login flow with no error-handling requirement was the most common case we ran into.

RequirementImprover handles underspecified statements. "The system should respond quickly" becomes "the system shall respond within 1 second of form submission." Each module assigns a severity rating — low, medium, or high — and those ratings carry forward into artifact prioritization.

The LLM modules write their results to an AnalysisReport. The generation layer reads that and produces four output types.

The primary output is an SRS structured according to ISO/IEC/IEEE 29148 [4]. Eleven sections get populated from AnalysisReport fields: Introduction, Scope, Overall Description, Functional Requirements, User Characteristics, Constraints, Assumptions and Dependencies, Data Requirements, External Interface Requirements, Quality Attributes, and Detected Conflicts.

We render the SRS to PDF with fpdf2, mostly because it handles Turkish characters out of the box without extra setup. In the Functional Requirements section, each requirement appears as one row in a three-column table: identifier, text, and priority. The DOCX path is separate and only covers user stories. That one uses python-docx.

Functional requirements turn into user stories with the standard "As a [actor], I want to [action], so that [benefit]" template. A dedicated LLM call produces all three fields. It takes the requirement text together with its improvement record, and the NER and dependency-parse outputs come in as context rather than getting slotted mechanically into the template. Priority follows the severity rating of any linked analysis finding.

Fig. 3. Transformation from a parsed requirement to a user story and Gherkin BDD scenario.

Gherkin BDD scenarios come out of a separate LLM call. It pulls the functional requirements straight from the AnalysisReport and emits one Given/When/Then block per requirement. Preconditions, inputs, and acceptance criteria draw on the improvement records included in the prompt, so the measurable criteria RequirementImprover added earlier carry through to the test scenarios without anyone re-typing them. The .feature files come out formatted for direct import into standard test automation frameworks.

The fourth output is a product backlog in XLSX format. Each item inherits its priority from the severity rating of its corresponding user story, keeping traceability intact from the original requirement through to the sprint artifact.

IV. RESULTS AND EVALUATION

Fig. 4. AutoReq results interface showing detected conflicts with severity ratings and requirement pairs.

Evaluation used two corpora. The development corpus had 63 sentences across five domains: e-commerce, online banking, appointment scheduling, corporate HR, and mobile application. A separate 30-sentence healthcare corpus was held out from all development and optimization work and used only for generalization testing. Both were adapted from real-world stakeholder materials.

Annotation was done by the primary author and cross-validated using Gemini and Claude as independent labelers. Disagreements were resolved by author judgment. Not ideal, but there are no established Turkish requirements benchmarks to fall back on. All experiments ran on the same hardware using Gemini 2.5 Flash via OpenRouter.

Development corpus. Actor extraction came in at 56.1% precision and 82.2% recall. Precision took a hit from false positives in the mobile domain, where noun phrases like "sayfa geçişleri" and "çevrimdışı mod" got picked up as actors — the lemma table covers roles but has no filter for non-human noun phrases that surface in NER position. FR/NFR classification reached 93.7% with the hybrid approach. ConflictDetector identified 100.0% of ground-truth conflict pairs (6/6) with a 0.0% false positive rate. That is a big jump from the earlier 85.7% detection at 11.3% false positives. We attribute it to confidence-based post-processing (threshold=0.6) and pairwise deduplication filtering. A notable portion of those earlier false positives came from the model treating optional-versus-mandatory field distinctions as logical contradictions — technically defensible, but not the kind of finding an engineer would actually act on. We think that is a prompt-level issue, not an architectural one.

Healthcare generalization. On the held-out healthcare corpus, FR/NFR accuracy came in at 96.7%, actor precision at 51.4%, and actor recall at 90.5% (F1=65.5%). The precision drop relative to the development corpus is expected: domain-specific terms like "hasta" appear both as actors and as modifiers in non-actor contexts (e.g. "hasta verilerinin yedeklenmesi"), generating false positives that the lemma-based system does not filter. The high recall, however, confirms that the extended lemma table and dependency-parse extraction generalize well to unseen healthcare vocabulary. FR/NFR classification improved markedly over the development corpus, suggesting that the three-layer classifier handles domain-specific NFR patterns (KVKK compliance, SLA targets, medical data encryption) robustly.

Ablation. Disabling all three LLM modules drops conflict detection to 0%, which confirms the obvious point that the NLP layer alone cannot perform logical analysis across a requirement set. At the classification layer, the hybrid system averages 1.15s per sentence versus 0.68s for the NLP-only path—a 1.69× slowdown. The LLM fallback (Layer 3) was triggered for 25% of sentences (5 of 20); those sentences averaged 2.45s each, while sentences resolved by Layers 1 and 2 averaged 0.72s. Across all sentences the average LLM overhead is +0.47s per sentence. Latency variance is driven by API round-trip time rather than internal processing. For single-document use, this is not a meaningful obstacle. Batch processing is another matter and would benefit from response caching or asynchronous calls; the parallel thread structure already in place helps somewhat.

A few things worth flagging. The 63-sentence development corpus is small — the reported figures are indicative, not definitive. A larger evaluation against something like the PROMISE dataset or a relevant OpenScience subset would give a more reliable picture.

The system only supports Turkish. Adding another language means swapping in the right Stanza model and rebuilding the lexical resources from scratch.

The FR/NFR classifier's LLM fallback adds latency per sentence. Replacing it with a locally trained model is the next planned step.

V. CONCLUSION

AutoReq is, to our knowledge, the first end-to-end pipeline that takes Turkish stakeholder text as input and produces a complete set of requirements engineering artifacts—SRS, user stories, Gherkin scenarios, product backlog—without manual intervention between stages. The hybrid architecture connects Stanza-based morphological preprocessing to two parallel LLM analysis modules, ConflictDetector and GapAnalyzer, and then a generation stage where RequirementImprover runs at the same time as the four artifact generators. On the development corpus we obtained 100.0% contradiction detection with 0% false positives, 82.2% actor recall, and 93.7% FR/NFR classification. On the held-out healthcare corpus, FR/NFR accuracy reached 96.7% and actor recall 90.5%, which suggests both the NLP and LLM layers generalize well beyond the development domains.

The system makes a few contributions worth summarizing. The most important is probably the conflict detection component itself—not because conflict detection is new, but because recording conflict type, requirement identifiers, and a natural-language rationale for each detected pair actually gives analysts something useful to work with, rather than a list of flagged pairs to investigate from scratch. The unified generation layer is also worth mentioning; four artifact types from a single analysis pass means avoiding the kind of integration mess that tends to build up in multi-tool pipelines.

The language-specific NLP configuration doesn't port easily. Rebuilding the lexical resources alone is a significant effort, and that's before touching the Stanza setup for a new language.

External API calls add latency. At batch scale that becomes a real problem — the parallel thread structure helps, but only somewhat.

The evaluation corpus is small enough that the numbers carry real uncertainty. We've tried to be upfront about that throughout the paper.

Planned next steps: a domain-specific Turkish NER model to replace the lemma-based approach, English and German support, Jira and Azure DevOps integration, and usability studies with actual requirements analysts.

That last item is probably the most important one. It's also the one that keeps getting pushed back.
