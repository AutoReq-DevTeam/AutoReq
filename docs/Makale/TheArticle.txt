
Abstract—Requirements defects, as a root cause of software project failure, have received a remarkable amount of attention relative to how little the situation has actually improved. Industry surveys going back twenty-plus years consistently attribute somewhere around 40% of project failures to poorly written or entirely missing requirements specifications, and yet the process by which stakeholder interviews get translated into unambiguous, verifiable documents has not changed in any fundamental sense for most organizations. This paper describes AutoReq, a pipeline we developed to try to automate this translation for Turkish-language input—we chose Turkish specifically because most existing tools simply do not support it, which is a gap that is rarely acknowledged. The system works in three stages: a Stanza-based preprocessing module handles sentence segmentation, actor and business object extraction, and FR/NFR labeling; a two-stage LLM analysis layer where ConflictDetector and GapAnalyzer run first in parallel over Gemini 2.5 Flash via OpenRouter and RequirementImprover kicks in alongside the artifact generators in a second parallel step; and a generation layer at the end that assembles an ISO/IEC/IEEE 29148-compliant SRS, user stories, Gherkin BDD scenarios, and a product backlog. Results on a 63-sentence development corpus covering five domains showed 56.1% precision and 82.2% recall for actor extraction, 93.7% FR/NFR classification accuracy, and 100.0% contradiction detection with 0% false positives. On a separately held-out 30-sentence healthcare corpus, FR/NFR accuracy reached 96.7% and actor recall 90.5%. Expert raters scored the generated SRS documents at an average of 4.34 out of 5. The numbers are reasonable for a first system of this kind, though we would not want to overstate them.

Keywords— Software Requirements Engineering, Natural Language Processing, Large Language Model, Automated Document Generation, Software Requirements Specification, Contradiction Detection, Behavior-Driven Development, Requirements Classification

I. Introduction

Software projects fail in many ways, and we are not going to pretend that requirements are the only problem, but they do appear in post-mortems with a consistency that is somewhat difficult to ignore. The Standish Group's CHAOS Report puts requirements-related failures at around 40% of all project failures [1]—a figure that has been cited so many times it has almost lost its impact, but the underlying observation is still valid. Boehm and Basili [2] showed, back in 2001, that catching a defect during the requirements phase costs roughly one tenth of what the same defect costs if it reaches implementation, and substantially less than what it costs if it reaches production. These numbers have been known for a long time. The practice has not changed much.

Part of why this remains so persistent is structural. Contradictions in a requirements document rarely look like "X must happen and X must not happen" sitting next to each other. They emerge from interactions between requirements written at different times, often by different people responding to different pressures. A requirement saying "the system should be fast" passes review without anyone flagging it—and then causes real problems when developers sit down to implement it and realize nobody ever said what fast actually means in this context [3]. Wagner et al. [3] looked at this pattern across multiple countries and organization types and found it everywhere. It is not a sign of immature teams. It is a structural property of the elicitation process itself—which is, honestly, not a comfortable thing for the field to sit with.

Various tools have tried to address portions of this. Rule-based NLP handles entity extraction reasonably well under constrained conditions. Classification models have been applied to FR/NFR separation with mixed results. LLM-based systems can rewrite ambiguous requirements, sometimes convincingly, though they are not generally designed to detect logical contradictions across a full requirement set or enforce any specific document structure. To our knowledge—and we acknowledge we may have missed something—no currently published tool takes unstructured stakeholder text as input and produces ISO/IEC/IEEE 29148-compliant output in a single automated pass, without requiring manual intervention somewhere in the middle [4]. AutoReq was built to address that.

The system passes stakeholder text through three layers: morphological analysis and dependency parsing via Stanza, concurrent LLM analysis using Gemini 2.5 Flash over OpenRouter with JSON-constrained outputs, and a generation layer that produces four artifact types from a single analysis pass. We describe the architecture, the evaluation methodology, and the results in the sections that follow.

What makes AutoReq different from prior work, at least in our view, comes down to a few things. The NLP–LLM integration handles the entire pipeline without any manual handoffs between stages, which sounds straightforward but turns out to require more engineering effort than one might expect. The ConflictDetector does not just flag which requirements conflict—it records what kind of conflict is present and why, in natural language, which is the information an analyst actually needs to do something with the finding. All four output artifact types come out of a single analysis pass—no separate processing runs for each format.

The rest of the paper covers related work in Section II, system architecture in Section III, evaluation results in Section IV, and conclusions in Section V.

II.  Lıterature Review

Automated extraction of requirements from natural language text is not a new problem. It has been studied within NLP for quite some time, with varying degrees of success and, one has to say, varying degrees of practical applicability.

Dalpiaz et al. [6] combined interpretable machine learning with dependency parse features and reported FR/NFR classification accuracy above 85%, which is good. The caveat is that their approach was designed for English, and it does not transfer well to Turkish. Turkish is agglutinative—a single root can appear in dozens of inflected surface forms depending on morphological context, which breaks the feature assumptions underlying most English-trained classifiers in ways that are not easy to patch. Ferrari and Esuli [7] tackled ambiguity at the lexical level across domains, which is a useful contribution, but their method does not deal with contradictions that only emerge when two non-adjacent sentences are read against each other. That particular problem has received less attention than it probably deserves.

LLM-based approaches to software engineering tasks have proliferated, though not evenly. Fan et al. [8] note in their survey that code generation and bug localization dominate the literature, while requirements engineering specifically remains underexplored. Ronanki et al. [9] examined LLM-assisted elicitation and found a pattern that, once you see it, you see it everywhere: the models generate plausible-sounding clarifying questions but also introduce hallucinated requirements and structurally inconsistent outputs with some regularity. The implication is straightforward—an LLM used without structured preprocessing and output validation is not a reliable component for requirements work. It can be useful inside a more constrained pipeline, but it should not be the pipeline.

Automated SRS generation as a standalone research problem has received surprisingly little attention, given how much effort went into specifying what these documents should actually contain. ISO/IEC/IEEE 29148 [4] lists the required sections—scope statements, stakeholder definitions, use cases, formal requirement tables—but most tools that claim to produce SRS output either fill those sections from rigid templates or require analysts to enter content manually. We could not find a published system that derives section content automatically from an intermediate analytical representation of the requirements. There may be commercial tools that do this. If so, they are not described anywhere we could access.

Contradiction detection has a longer history, rooted in formal logic and ontology-based reasoning. Guo et al. [10] report 83.9% accuracy on pairwise conflict detection using semantic analysis. Their system identifies conflicting pairs, which is useful. The problem is that it assigns the same generic conflict label regardless of whether the issue is a direct logical contradiction, a scope overlap, or an implied missing precondition. For someone actually trying to fix the conflict, those distinctions matter quite a bit—the corrective actions are completely different.

BDD scenario generation from requirements is a more recent area. Ronanki et al. [11] evaluated ChatGPT specifically on user story quality and found substantial variation in output consistency—the generated stories needed expert review before anyone would trust them in a real workflow. This is consistent with the elicitation findings. Output quality downstream is always bounded by how well the upstream extraction step worked, and this seems underappreciated in a lot of the recent LLM-in-SE literature.

The overall picture is a field where individual pipeline components have been studied, sometimes quite thoroughly, but no integrated system connecting extraction, classification, contradiction detection, and scenario generation has been published. AutoReq is an attempt to build that integration—with the added constraint of supporting Turkish, which is non-trivial given the language's morphological complexity.

III. System Architecture and Methodology

AutoReq has three layers. Stakeholder text goes in at the top, hits the preprocessing layer first, then gets forwarded to the LLM analysis layer, and whatever comes out of that is handed to the generation layer to produce the final documents. Early on we decided that all data moving between layers had to go through exactly two typed classes in core/models.py—ParsedDocument carries preprocessing output, AnalysisReport carries LLM analysis output. Nothing can show up as input to a downstream layer unless something upstream explicitly wrote it there. This turned out to be one of the better decisions we made; it caught a surprising number of integration bugs early. The whole thing runs in Python, with a Streamlit front-end that calls a central process_text() function.

Fig. 1. Three-layer pipeline architecture of AutoReq.

Preprocessing. The preprocessing layer runs the Turkish Stanza model [5], applying tokenization, multi-word tokenization (MWT), POS tagging, lemmatization, and NER to each input text. Memory consumption is approximately 300 MB; @st.cache_resource prevents the model from being reloaded on repeated calls. Sentence boundary detection uses Stanza's built-in segmenter, and each sentence gets a unique identifier in the REQ_001 format.

Actor extraction works via lookup against a predefined lemma table. If a sentence's lemma sequence contains any entry from the set {kullanıcı, sistem, yönetici, admin, müşteri}, the corresponding actor is recorded; business objects use an analogous table. Working at the lemma level is not optional for Turkish—surface-level matching produces counts that are meaninglessly inflated given the morphological variation. Stanza failing to initialize was more of a problem during development than we expected, so a substring-matching fallback handles extraction when that happens. Initial testing revealed coverage gaps, so we extended the lemma table to include roles common in healthcare, education, logistics, and legal domains. We also added a secondary extraction pass using Stanza dependency trees—nsubj-tagged nouns get treated as additional actor candidates, though we filter out system-level tokens like 'uygulama' and 'platform' when a human role is already present in the sentence.

FR/NFR classification uses a three-layer hybrid approach. A functional-verb detector screens for action-oriented predicates first; an NFR keyword-and-regex matcher then checks for performance, security, latency, SSL, and related indicator terms; sentences unresolved by both layers get forwarded to the Gemini LLM for a final classification call. The earlier keyword-only baseline had a frustrating failure mode—terms like kullanıcı and şifre were incorrectly triggering NON_FUNCTIONAL labels despite appearing constantly in functional contexts—and that alone dragged accuracy down to 69.8%. The hybrid system brings it up to 93.7%.

LLM Analysis. The ParsedDocument goes first to two LLM modules running concurrently inside a ThreadPoolExecutor with max_workers=2: ConflictDetector and GapAnalyzer. RequirementImprover runs in a second ThreadPoolExecutor with max_workers=6, where it shares the pool with the artifact generators. Each module serializes the requirement list in the format [REQ_NNN] (TYPE) TEXT, wraps it in a two-part prompt—a system message defining the module's role and a user message specifying the requirement list and expected JSON schema—and submits it to the Gemini 2.5 Flash API. The API occasionally returns markdown-formatted text instead of bare JSON, which we did not anticipate and was genuinely annoying to deal with, so all three modules pass their responses through an extract_json_object() utility that strips formatting before parsing.

Fig. 2. Example input–output of the LLM analysis layer: conflict detection and optimization results.

ConflictDetector sorts detected pairs into one of six buckets: logic, business_rule, performance, security, usability, or other. Logic is straight contradiction—one requirement asking for a password reset while another forbids password changes. business_rule covers two requirements that hit the same workflow with incompatible parameters. performance, security, and usability narrow conflicts down to those specific non-functional areas—contradictory latency targets, authentication policies that don't align, or accessibility constraints that conflict with each other. Pairs that don't fit any of those five categories land in other. GapAnalyzer looks for flows the requirement set implies but never actually states—a login flow with no error-handling requirement is the most common example we encountered. RequirementImprover handles underspecified statements; "the system should respond quickly" gets rewritten to "the system shall respond within 1 second of form submission." Each module assigns a severity rating of low, medium, or high, and those ratings carry forward into artifact prioritization.

Output Generation. The three LLM modules write their results to an AnalysisReport, which the generation layer reads to produce four output types.

The primary output is an SRS structured according to ISO/IEC/IEEE 29148 [4], with eleven sections populated from AnalysisReport fields: Introduction, Scope, Overall Description, Functional Requirements, User Characteristics, Constraints, Assumptions and Dependencies, Data Requirements, External Interface Requirements, Quality Attributes, and Detected Conflicts. For PDF output we used fpdf2, mainly because it handles Turkish characters correctly without any additional configuration; DOCX output uses python-docx and covers user stories. Each requirement in the Functional Requirements section appears as one row in a three-column table: identifier, text, and priority.

User stories are generated for functional requirements using the "As a [actor], I want to [action], so that [benefit]" template via a dedicated LLM call. The NER and dependency-parse outputs come in as context rather than being slotted mechanically into the template fields. Priority matches the severity rating of any linked analysis finding.

Fig. 3. Transformation from a parsed requirement to a user story and Gherkin BDD scenario.

Gherkin BDD scenarios come from a separate LLM call that pulls functional requirements directly from the AnalysisReport and emits one Given/When/Then block per requirement. Preconditions, inputs, and acceptance criteria are pulled from the improvement records, so whatever measurable criteria RequirementImprover introduced earlier shows up in the test scenarios without any manual re-entry. The .feature files are formatted for direct import into standard test automation frameworks. The fourth output is a product backlog in XLSX format. Each item inherits its priority from the severity rating of its corresponding user story, which keeps traceability intact from the original requirement all the way through to the sprint artifact.
 
IV. Results and Evaluatıon

Fig. 4. AutoReq results interface showing detected conflicts with severity ratings and requirement pairs.

Evaluation used two corpora. The development corpus comprised 63 sentences across five domains—e-commerce, online banking, appointment scheduling, corporate HR, and mobile application. A separate 30-sentence healthcare corpus was held out from all development and optimization work, used only for generalization testing. Both corpora were adapted from real-world stakeholder materials. Annotation was carried out by the primary author and cross-validated using Gemini and Claude as independent labelers, with disagreements resolved by author judgment—not ideal, but there are no established Turkish requirements benchmarks to fall back on. All experiments used the same hardware and Gemini 2.5 Flash via OpenRouter.

Development corpus. Actor extraction yielded 56.1% precision and 82.2% recall. Precision took a hit from false positives in the mobile domain, where noun phrases like "sayfa geçişleri" and "çevrimdışı mod" were picked up as actors—the lemma table covers roles but has no filter for non-human noun phrases that surface in NER position. FR/NFR classification reached 93.7% with the hybrid approach. ConflictDetector identified 100.0% of ground-truth conflict pairs (6/6) with a 0.0% false positive rate. This is a substantial jump from the earlier 85.7% detection at 11.3% false positives; we attribute it to confidence-based post-processing (threshold=0.6) and pairwise deduplication filtering. A notable portion of those earlier false positives came from the model treating optional-versus-mandatory field distinctions as logical contradictions—technically defensible, but not the kind of finding an engineer would actually act on. We think this is a prompt-level issue, not an architectural one. Expert ratings of the generated SRS documents averaged 4.34 out of 5 for completeness and structural readability, across three independent software engineering researchers.

Healthcare generalization. On the held-out healthcare corpus, FR/NFR accuracy came in at 96.7%, actor precision at 51.4%, and actor recall at 90.5% (F1=65.5%). The precision drop relative to the development corpus is expected: domain-specific terms like "hasta" appear both as actors and as modifiers in non-actor contexts (e.g. "hasta verilerinin yedeklenmesi"), generating false positives that the lemma-based system does not filter. The high recall, however, confirms that the extended lemma table and dependency-parse extraction generalize well to unseen healthcare vocabulary. FR/NFR classification improved markedly over the development corpus, suggesting that the three-layer classifier handles domain-specific NFR patterns (KVKK compliance, SLA targets, medical data encryption) robustly.

Ablation. Disabling all three LLM modules drops conflict detection to 0%, which confirms the obvious point that the NLP layer alone cannot perform logical analysis across a requirement set. At the classification layer, the hybrid system averages 1.15s per sentence versus 0.68s for the NLP-only path—a 1.69× slowdown. The LLM fallback (Layer 3) was triggered for 25% of sentences (5 of 20); those sentences averaged 2.45s each, while sentences resolved by Layers 1 and 2 averaged 0.72s. Across all sentences the average LLM overhead is +0.47s per sentence. Latency variance is driven by API round-trip time rather than any internal processing. For single-document use, this is not a meaningful obstacle. Batch processing is another matter and would benefit from response caching or asynchronous calls; the parallel thread structure already in place helps somewhat.

A few limitations bear mentioning. The 63-sentence development corpus is small, and the reported figures should not be read as definitive performance estimates—they are indicative. Larger evaluations using something like the PROMISE dataset or a relevant OpenScience subset would give a more reliable picture. The system supports only Turkish; extending to other languages would require Stanza language models and rebuilt lexical resources for each. The FR/NFR hybrid classifier's LLM fallback layer adds latency per sentence; replacing it with a locally trained model is next on the list.

V. Conclusion

AutoReq is, to our knowledge, the first end-to-end pipeline that takes Turkish stakeholder text as input and produces a complete set of requirements engineering artifacts—SRS, user stories, Gherkin scenarios, product backlog—without manual intervention between stages. The hybrid architecture connects Stanza-based morphological preprocessing to two parallel LLM analysis modules (ConflictDetector and GapAnalyzer), with RequirementImprover running in a second parallel stage alongside the four artifact generators. On the development corpus we obtained 100.0% contradiction detection with 0% false positives, 82.2% actor recall, and 93.7% FR/NFR classification. On the held-out healthcare corpus, FR/NFR accuracy reached 96.7% and actor recall 90.5%, which suggests both the NLP and LLM layers generalize well beyond the development domains.

The system makes a few contributions worth summarizing. The most important is probably the conflict detection component itself—not because conflict detection is new, but because recording conflict type, requirement identifiers, and a natural-language rationale for each detected pair actually gives analysts something useful to work with, rather than a list of flagged pairs to investigate from scratch. The unified generation layer is also worth mentioning; four artifact types from a single analysis pass means avoiding the kind of integration mess that tends to build up in multi-tool pipelines.

There are real limitations. The language-specific NLP configuration does not port easily to other languages—rebuilding the lexical resources alone would be a significant effort. External API calls introduce latency that will become a real problem at batch scale, though the parallel thread structure helps somewhat. And the evaluation corpus is small enough that the numbers carry meaningful uncertainty; we have tried to be upfront about this throughout the paper.

Future plans include training a domain-specific Turkish NER model to replace the lemma-based approach, extending support to English and German, integrating with Jira and Azure DevOps, and running usability studies with actual requirements analysts to see how the system affects their workflows in practice. That last item is probably the most important one, and, predictably, the one that keeps getting pushed back.
